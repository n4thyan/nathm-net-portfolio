AI Workflow

How I use AI-assisted development to plan, build, debug and improve practical projects while keeping the final work human-reviewed and honestly documented.

Working method

AI-assisted builder, not a traditional software engineer

I am not presenting myself as an experienced software engineer. I am a self-taught practical builder using AI-assisted workflows to create small web tools, dashboards, automation projects and local AI experiments.

I understand what I want a project to do, how the main pieces should work together and what the finished behaviour should look like. I use AI to help with implementation details, debugging, refactoring, documentation and learning around the code.

Web design and development are areas where this workflow is especially useful for me. I can start from a rough idea, work with AI to plan the structure, build the page or app, test it locally, fix issues and improve the public-facing wording until the project is clear enough to explain.

Planning

Turning rough ideas into project structure, feature lists, implementation steps and realistic milestones.

build

Drafting code, comparing approaches, explaining unfamiliar sections and helping connect frontend, backend and data files.

Troubleshooting

Documentation

Improving README files, project notes, public wording, setup steps and honest status sections.

Review

Learning

Using each project to slowly improve my understanding of web development, JavaScript, Python, Node.js, APIs and deployment.

Context matters

Why I work from repositories

Basic copy-pasting into a chatbot can help with small errors, but it often misses the wider project context. I get better results when the AI can work from the repository, file structure, README, previous decisions, terminal output and the actual problem I am trying to solve.

This matters because real projects are not just isolated snippets. They include configuration, assets, deployment paths, documentation, naming, dependencies and user-facing wording. Using GitHub as the source of truth gives the AI a much clearer picture of the whole project.

repo_context

  • File structure visible
  • README and project status visible
  • Changes can be reviewed through commits
  • Better than isolated snippets
  • Still needs human checking

02. What I still check myself

I do not treat AI output as automatically correct. Before I count something as finished, I try to test the visible behaviour, check the public wording, run the project locally where possible and document what has or has not been validated.

Some parts of the code are still above my current level. I am open about that. My goal is to use AI responsibly as a learning and building tool, not to pretend that I understand every line at a professional level.

I also know that higher-risk work needs more care. Anything involving payments, private data, authentication, moderation, live streaming or public deployment needs clearer testing, stronger limits and more human review.

  • Local testing
  • Visible behaviour
  • Setup steps
  • README accuracy
  • Public wording
  • Known limitations
  • Deployment checks
  • Human review

Why I think it is viable

AI is a practical tool, not just a chatbot

AI-assisted development is valuable to me because it helps close the gap between an idea and a working prototype. It can speed up planning, explain code, suggest fixes, generate first drafts and help me learn by working through real problems.

I do not claim to get more out of AI than an experienced traditional developer. My point is simpler: I know how to use modern AI tools effectively for my level, and each project helps me build more structure, confidence and technical understanding over time.

I also experiment with both cloud AI tools and local models. Local models can be useful for private workflows, stream tools and projects where I want more control over the setup, even if performance depends on the hardware available.

clear_limits

  • Not claiming senior or junior developer status
  • Not claiming AI replaces experienced developers
  • Not treating generated code as automatically safe
  • Using AI as a practical workflow and learning tool
